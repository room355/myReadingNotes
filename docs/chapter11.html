Multiple consumers

Two main patterns:

  • Load balancing
    • Each message is delivered to one of the consumers, the consumers share the work of processing the messages in the topic.
  • Fan-out
    • Each message is delivered to all of the consumers

Two pattern can work together:
Two separate groups of consumers may each subscribe to a topic, such that each group collectively receives all messages, but within each group only one of the nodes receives each message. (Kafka)

Acknowledgments and redelivery

To avoid messages lost, acknowledgments are used:

  • A client must explicitly tell the broker when it has finished processing a message so that the broker can remove it from the queue.
  • Lost messages will be delievered again if the ack is not received by broker

Effect:
THe combination of load balancing with redelivery leads to messages being reordered.

Partitioned Logs

log-based message brokers:
Combing the durable storage approach of databases with the low-latency notification facilities of messaging.

Using logs for message storage

partition
In order to scale to higher throughput, the log can be partitioned

  • Different partition can be hosted on different machines
  • Making each partition a separate log that can be read and written independently from other partitions.

offset

  • A monotonically increasing sequence number
  • Marks the processing of messages
Logs compared to traditional messaging

Log-based messaging:

  • Support fan-out messaging
  • Assign entire partitions to nodes in the consumer group

Each client consumes all the messages in the partitions it has been assigned and it reads the messages in the partition sequentially.

  • The number of nodes sharing the work of consuming a topic can be at most the number of log partitions in that topic, because messages within the same partition are delivered to the same node
  • If a single message is slow to process, it holds up the processing of subsequent messages in that partition
Consumer offsets

Make it easy to track which messages have been processed.
It starts consuming messages at the last recorded offset.

Disk space usage

Circular buffer or Ring buffer:
The log implements a bounded-size buffer that dis cards old messages when it gets full

Need to consider the limit of the buffer

When consumers cannot keep up with producers

If a consumer does fall too far behind and starts missing messages, only that consumer is affected and does not disrupt the service for other consumers.

A big operational advantage:
you can experimentally consume a production log for development, testing, or debugging purposes, without having to worry much about disrupting production services.

Replaying old messages

Offset is controlled by consumer thus messages can be replayed at will.

Databses and Streams

state machine replication:
If every event represents a write to the database, and every replica processes the same events in the same order, then the replicas will all end up in the same final state.

Keeping Systems in Sync

If only one replicated database with a single leader, then that leader determines the order of writes. - Make state machine replication easier.

Change Data Capture (CDC)

The process of observing all data changes written to a database and extracting them in a form in which they can be replicated to other systems.

Implementation
CDC is a mechanism for ensuring all changes made to the system of record are also reflected in the derived data systems so that the derived systems have an accurate copy of the data.

  • Can be implemented by log-based message broker
  • Database trigger
    • Performance impact
    • Unstable
  • CDC is usually asyn

Question:
Cant track all changes due to limited resource

Solution:
Log compaction

Log compaction
A compacted log depends only on the current contents of the database, not the number of writes that have ever occurred in the database.

  • If the CDC system is set up such that every change has a primary key, and every update for a key replaces the previous value for that key, then its sufficient to keep just the most recent write for a particular key
Event Sourcing

Event sourcing involves storing all changes to the application state as a log of change events.
Event Sourcing vs CDC

Event Sourcing CDC
Storage The application logic built on the basis of immutable events that are written to an event log The application uses the database in a mutable way, changes at will
Implementation Level Reocrds events happened at the application level Record log of changes at a low level

Idea of Event sourcing:
It is more meaningful to record the users actions as immutable events rather than the effect of these actions

Deriving current state from the event log
  • Take the log of events and transform it into application state
  • Storing snapshots of the current state that is derived from the log of events

State, Streams, and Immutability

Advantage of immutable events
  1. Provides auditability
  2. Capture more information than just the current state
Deriving several views from the same event log

target:
Derive several different read-oriented representations from the same log of events.

  • Gain flexibilty by separating different read views (command query responsibility segregation)
Concurrency

Down side of Event sourcing and CDC is that consumers of the event log are usually asynchronous.
Solution of dealing with asyn data:

  • Perform the updates of the read view synchronously with appending the event to the log.
  • Deriving the current state from an event log simpifies concurrency control.

Timing issues

Event time is different form processing time

  • E.X. the processor shut down and we lost correct rating on any measurment on processing time.

To handle this, we need to define window for events.
stragler events: events that got delayed and arrived after the boundary of the window
To handle this:

  • Ignore the straggler events
  • Publish a correction, an updated value for the window with stragglers included, may need to retract the prvious output

To adjust incorrect device clocks:

  • The time at which the event occurred, according to the device clock
  • The time at which the event was sent to the server, according to the device clock
  • The time at which the event was received by the server, according to the server clock

By subtracting the second timestamp from the third, you can estimate the offset between the device clock and the server clock.
You can then apply that offset to the event timestamp, and thus estimate the true time at which the event actually occurred.

Types of windows
  • Tumbling window
    • A tumbling window has a fixed length, and every event belongs to exactly one window.
  • Hopping window
    • A hopping window also has a fixed length, but allows windows to overlap in order to provide some smoothing.
  • Sliding window
    • A sliding window contains all the events that occur within some interval of each other.
  • Session window
    • Unlike the other window types, a session window has no fixed duration.

Fault Tolerance

Solutions:

  • Microbatching
    • Break the stream into small blocks, and treat each block like a miniature batch process.
  • checkpointing
    • Periodically generate rolling checkpoints of state and write them to durable storage. If a stream operator crashes, it can restart from its most recent checkpoint and discard any output generated between the last checkpoint and the crash.
  • Atomic commit revisited
    • Managing both state changes and messaging within the stream processing framework.
