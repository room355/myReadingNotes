Chapter 10 Batch Processing

Different types of system:

  • Services (online systems)
    • Input: request or instruction
    • When one request or instruction is received, the service tries to handle it as quickly as possible and sends a response back.
    • Performance Measurement: Response time
  • Batch Processing systems (offline systems)
    • Input: fixed set of input data
    • A batch processing system takes a large amount of input data, runs a job to process it, and produces some output data.
    • Performance Measurement: throughput (the time it takes to crunch through an input dataset of a certain size)
  • Stream processing systems(near-real-time systems)
    • Input: events, messages
    • Consumes inputs and produces outputs, a stream job operates on events shortly after they happen
    • Performance Measurement: latency (maybe)

One implementation is HDFS:

  • Based on shared-nothing principle
  • Consists of a daemon process running on each machine, exposing a network service that allows other nodes to access files stored on that machine
  • A central server called the NameNode keeps track of which file blocks are stored on which machine
  • In order to tolerate machine and disk failures, file blocks are replicated on multiple machines
MapReduce Job Execution

MapReduce is a programming framework with which you can write code to process large datasets in a distributed filesystem like HDFS.

Pattern of MapReduce:

  1. Read a set of input files, and break it up into records.
  2. Call the mapper function to extract a key and value from each input record.
  3. Sort all of the key-value pairs by key.
  4. Call the reducer function to iterate over the sorted key-value pairs. If there are multiple occurrences of the same key, the sorting has made them adjacent in the list, so it is easy to combine those values without having to keep a lot of state in memory.

Mapper:

  • Called once for every input record and its job is to extract the key and value from the input record.
  • Doesnt keep any state from one input record to the next, so each record is handled independently.

Reducer:

  • Take k-v pair produced by the mappers, collect all the values belonging to the same key
  • Call the reducer with an iterator over that collection of values
Distributed execution of MapReduce

MapReduce parallelization:
Partitioning:

  • Input: A directory in HDFS
  • Partition: Each file or file block within the input directory is considered to be a separate partition
  • Partition can be processed by a separate map task (m1, m2)

Scheduler:

  • The scheduler tries to run each mapper on one of the machines that stores a replica of the input file
  • Putting the computation near the data: run the map task locally
    • Saves copying files over network

Mapping:

  1. The number of map tasks is determined by the number of input file blocks
  2. Copy the code (e.g., JAR files) to the appropriate machines
  3. Start the map task and begins reading the input file
  4. Passing one record at a time to the mapper callback
  5. The output consists of k-v pairs

Sorting:

  1. Performed in stages
  2. Each map task partions its output by reducer, based on the hash of the key
  3. Each of these partitions is written to a sorted file on the mappers local disk

Reducing:

  1. The number of reduce tasks is configured by the job author
  2. The framework uses a hash of the key to determine which reduce task should receive a particular k-v pair
  3. k-v pairs must be sorted
  4. Cconnect to each of the mappers and download the files of sorted k-v pairs for their partition
  5. The reduce task takes the files from the mappers and merges them together, preserving the sort order
  6. The reducer is called with a key and an iterator that incrementally scans over all records with the same key

Whenever a mapper finishes reading its input file and writing its sorted output files, the MapReduce scheduler notifies the reducers that they can start fetching the output files from that mapper.

The shuffle
The process of partitioning by reducer, sorting, and copying data partitions from mappers to reducers.

MapReduce workflows
It is common for MapReduce jobs to be chained together so the output of one job becomes the input to the next job.

Advantange:
Using the MapReduce programming model has separated the physical network communication aspects of the computation from the application logic.
It also shields the application code from having to worry about partial failures, such as the crash of another node: MapReduce trans parently retries failed tasks without affecting the application logic.

Sort-merge joins

Secondary sort:
The MapReduce job can even arrange the records to be sorted such that the reducer always sees the record from the user database first, followed by the activity events in timestamp order

Figure example:

  1. The reducer can then perform the actual join logic easily: the reducer function is called once for every user ID, and thanks to the secondary sort, the first value is expected to be the date-of-birth record from the user database.
  2. The reducer stores the date of birth in a local variable and then iterates over the activity events with the same user ID, outputting pairs of viewed-url and viewer-age-in-years.

Sort-merge join:
Since the reducer processes all of the records for a particular user ID in one go, it only needs to keep one user record in memory at any one time, and it never needs to make any requests over the network.

GROUP BY
  1. The simplest way of implementing such a grouping operation with MapReduce is to set up the mappers so that the k-v pairs they produce use the desired grouping key.
  2. Another common use for grouping is collating all the activity events for a particular user session, in order to find out the sequence of actions that the user took. (sessionization)
    • Can be implemented by using cookie, user ID, or similar identifier
Handling skew

Like partition, hot keys also exists in MapReduce.

  • One reducer must process sigificantly more records than others
  • Any subsequent jobs must wait for the slowest reducer to finish

Solution:

  • Skew join
  • Sharded join

Broadcast hash joins

Broadcast reflects the fact that each mapper for a partition of the large input reads the entirety of the small input.
Hash reflects its use of a hash table.

Scenario:

  1. A large dataset is joined with a small dataset. In particular, the small dataset needs to be small enough that it can be loaded entirely into memory in each of the mappers.
  2. When a mapper starts up, it can first read the user database from the distributed filesystem into an in-memory hash table.
  3. The mapper can scan over the user activity events and simply look up the user ID for each event in the hash table
  4. There can still be several map tasks: one for each file block of the large input to the join. Each of these mappers loads the small input entirely into memory.
Partitioned hash joins

If the inputs to the map-side join are partitioned in the same way, then the hash join approach can be applied to each partition independently.

Map-side merge joins

If the input datasets are not only partitioned in the same way, but also sorted based on the same key.
In this case, it does not matter whether the inputs are small enough to fit in memory, because a mapper can perform the same merging operation that would normally be done by a reducer:

  • reading both input files incrementally, in order of ascending key, and matching records with the same key.

By treating inputs as immutable and avoiding side eggects,
batch jobs not only achieve good performance but also become much easier to maintain.

Beyond MapReduce

Intermediate state:
A means of passing data from one job to the next.

Materialization:
The process of writing out this intermediate state to files.

MapReduces fully materialization downsides:

  • Slow down the execution of the workflow as a whole since all tasks can only start once all previous tasks are done
  • Redundant mappers
  • Storing intermediate state may be overkill for temoprary files
Dataflow Engines
  • Handle an entire workflow as one job, rather than breaking it up into independent subjobs.
  • Explicitly model the flow of data through several processing stages

Operators:
Functions that dont take strict roles of alternating map and reduce, can be assembled in more flexible ways.

Fault tolerance:
The framework keep track of how given data was computed.
If the computation is deterministic:

  • Resume running an operator that ran into a fault during its execution

If not:

  • Kill the downstream operators and run them again on the new data.
Graphs and Iterative Processing

Iterative processing: Repeating until done

  1. An external scheduler runs a batch process to calculate one step of the algorithm.
  2. When the batch process completes, the scheduler checks whether it has finished.
  3. If it has not yet finished, the scheduler goes back to step 1 and runs another round of the batch process.

model:

  1. One vertex can send a message to another vertex, and typically those messages are sent along the edges in a graph.
  2. A function is called for each vertex, passing it all the messages that were sent to it.
  3. A vertex remembers its state in memory from one iteration to the next, so the function only needs to process new incoming messages. If no messages are being sent in some part of the graph, no work needs to be done.

Fault tolerance:

  • Pregel implementations guarantee that messages are processed exactly once at their destination vertex in the following iteration.
  • Fault tolerance is achieved by periodically checkpointing the state of all vertices at the end of an iteration.
High-Level APIs and Languages

Dataflow APIs generally use relational-style building blocks to express a computation: joining datasets on the value of some field; grouping tuples by key; filtering by some condition; and aggregating tuples by counting, summing, or other functions.

these high-level interfaces not only make the humans using the system more productive, but they also improve the job execution efficiency at a machine level.
